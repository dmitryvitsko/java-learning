# 📝 Пост и комментарии — toString(), геттеры и инкапсуляция

## 📋 Задача
Переопределить toString() у классов Post и PostComment
для текстового представления объектов.
Организовать доступ к приватным полям через геттеры.

---

## 🧠 Теория

### toString() — текстовое представление объекта
```java
System.out.println(post); // неявно вызывает post.toString()
```
Когда передаёшь объект в println() - Java автоматически
вызывает toString(). Без переопределения выведется что-то вроде
Post@3764951d - адрес в памяти. Бесполезно.

### Переопределение toString()
```java
@Override
public String toString() {
    return "Post{title='" + getTitle() + "', ...}";
}
```
@Override говорит что мы переопределяем метод из родителя Object.
Теперь println(post) выведет читаемый текст.

### Инкапсуляция - private поля и геттеры
```java
private String title; // закрыто снаружи

public String getTitle() { // открытый доступ на чтение
    return title;
}
```
Поле закрыто - изменить нельзя напрямую.
Геттер открыт - прочитать можно.
toString() использует геттеры а не поля напрямую.

### Вложенный toString()
```java
// В Post.toString() вызывается toString() каждого комментария
str += comments.get(i).toString();
```
Post содержит список PostComment. При выводе Post
автоматически вызывается toString() каждого комментария.
Это называется композиция объектов.

### Обработка null в whoLiked
```java
if (getWhoLiked() != null) {
    // выводим массив
} else {
    str += "null";
}
```
Если никто не поставил лайк - поле null.
Без проверки получим NullPointerException.

---

## 🛠 Что использовал
- `@Override toString()` - переопределение текстового представления
- `private` поля + геттеры/сеттеры - инкапсуляция
- Вложенный вызов toString() для комментариев
- Обработка null для необязательных полей
- `ArrayList<PostComment>` - список комментариев внутри поста
